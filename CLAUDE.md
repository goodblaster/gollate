# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Project Overview

**gollate** is a service that sorts OCR-extracted text blocks using canonical (ground truth) text. It takes unordered OCR output from various engines and returns blocks in their logical reading order by matching them against known text.

### Supported OCR Engines
- **tesseract** - Open-source OCR engine
- **easyocr** - Python-based OCR library
- **apple** - macOS Apple Vision framework (macOS only)

### OCR Regression Suite (the authoritative accuracy gate)

```bash
# Fast (~6s): sort committed OCR fixtures, score vs canonical, fail below baseline
go test ./integration          # or: make integration

# After a legitimate accuracy improvement, ratchet baselines up and commit
UPDATE_BASELINES=1 go test ./integration   # or: make baselines

# A/B test config changes: per-case accuracy delta table vs default config
./scripts/run-experiments.sh "Flag=value[,Flag=value...]" [...]

# Slow (minutes): regenerate documents + OCR data (when content/layouts change)
make build utils && ./scripts/generate-ocr-tests.sh

# Regenerate the seeded noise fixtures (only if their base fixtures change)
./scripts/generate-noise-fixtures.sh

# Optional: human-debugging artifacts (overlays, summary.txt, problems.todo)
./scripts/run-ocr-tests.sh
```

**Reading results:** accuracy is the percentage of canonical units (words;
characters for CJK) appearing in the output in the correct relative order
(LCS), so it punishes both missing text and wrong ordering. Per-case
baselines in `testdata/ocr-tests/baselines.json` only ratchet upward; a
failing case means a real regression. `*-noiseNN` fixtures measure
error tolerance (seeded misreads in NN% of blocks); `english-grid` measures
duplicate-short-line handling; the two Japanese vertical fixtures are
expected ~0% until vertical auto-detection exists (TESTING.md issue #4).

**Read TESTING.md first when working on tests or accuracy** — it holds the
current per-language scores, the diagnosed-but-unfixed algorithm issues
(pass-loop early exit, duplicate short-line anchoring, wrap filter, vertical
detection), and history that should not be relitigated (fuzzy matching was
removed deliberately). Suite mechanics: testdata/ocr-tests/README.md.

Key rules:

- **Language is the only external hint the sorter may receive** (mapped via
  `sorters.ConfigForLanguage`). Layout information — column counts,
  orientation, font sizes — must never be passed in; the algorithm may infer
  it internally from word-block geometry. `ConfigForLanguage` also enables
  the error-tolerance mechanisms each script measurably benefits from
  (wrap bridging + holes + anchoring + reconciliation for Latin/Hindi;
  anchoring only for Arabic; holes only for CJK) — these are algorithm
  strategy, not layout hints.
- Test documents are generated by `cmd/testdoc` (HTML/CSS via headless
  Chrome), which emits document.pdf, document.png (2x), and canonical.txt
  from the same source, and fails on page overflow. The old gofpdf tools
  (`cmd/text-to-pdf`, `cmd/chromepdf`, `cmd/blocks-to-pdf`) were removed.
- `canonical.txt` is the full page in reading order (title first, footer
  last). Accuracy is order-sensitive LCS (words; characters for CJK) against
  per-case baselines in `testdata/ocr-tests/baselines.json` that only ratchet
  upward.

## Architecture

### Core Algorithm

The sorting algorithm uses **pathfinding through permutations** to match OCR blocks to canonical text:

1. **Normalization** - Both canonical text and OCR blocks are normalized (lowercased, punctuation removed) for matching
2. **Line Sorting** - Canonical lines are processed longest-first to maximize accuracy
3. **Block Mapping** - OCR blocks are indexed by their normalized text content
4. **Path Finding** - For each canonical line, all possible paths through OCR blocks are explored
5. **Distance Calculation** - Paths are scored by spatial distance between consecutive blocks
6. **Optimal Path Selection** - The shortest (most likely) path is selected
7. **Block Removal** - Used blocks are removed from the pool to prevent reuse
8. **Post-Processing** - Remaining blocks are assembled into lines using spatial contiguity and punctuation

Key optimizations:
- **Rotation Optimization** - When exploring paths, blocks are searched starting from the most likely next position
- **Precurse** - For long lines, a limited-length search is performed first to find the optimal starting block
- **Permutation Limits** - Maximum permutations are capped to prevent exponential blowup
- **Shortest Path Pruning** - Paths exceeding the current shortest are abandoned early

### Package Structure

**`pkg/norm`** - Block normalization
- `Block` - Common normalized format for all OCR engines with bounding box, confidence, text
- Engine-specific normalizers convert from native formats (Tesseract, EasyOCR, Apple) to common `Block` format
- Bounding boxes are normalized to percentages of page dimensions

**`pkg/sorters`** - Core sorting algorithm
- `sorter.go` - Main Sorter struct containing input blocks, canonical lines, and mapping
- `sort.go` - Primary sorting implementation: the multi-pass loop, `recurse()` pathfinding, and `splitLine()` (the sentence/absent-word split cascade — measured still load-bearing on noisy pages)
  - `Sort()` - Primary entry point, performs multi-pass sorting with line splitting
  - `markIntermediateBlankLines()` - Preserves paragraph structure
  - `postSortBlocks()` - Final assembly of sorted output
- `sort_helpers.go` - Extracted helper functions for Sort() (NEW)
  - `buildChainForLine()` - Builds candidate block chain for each word
  - `shouldSkipLineInEarlyPass()` - Early pass line filtering logic
  - `initializePathfinding()` - Resets pathfinding state
  - `getPassPermutationLimit()` - Returns permutation limit for current pass
  - `shouldExitPassLoop()` - Determines multi-pass loop exit conditions
  - `shouldSkipChain()` - Validates chain length for pathfinding
  - `applyPrecurseOptimization()` - Applies precurse optimization to chain
  - `findBestPathsForLine()` - Performs pathfinding and sorts candidate paths
- `constants.go` - Algorithm constants (NEW)
  - `EarlyPassThreshold` - Maximum pass number for early-pass optimizations (3)
  - `MinChainLength` - Minimum words required for pathfinding (2)
  - `SingleWordChainLength` - Single-word chain identifier (1)
- `holes.go` - Error tolerance: wrap-bridging step admission and wildcard
  holes (missing words bridged by pathfinding, gap-fill claiming with
  exact-text confirmation)
- `linerepair.go` - Preparation step (on by default for Latin/Hindi):
  misread tokens identified by position within the engine's own line
  grouping and rekeyed to the canonical word; never changes Block.Text
- `anchor.go` - Context anchoring: duplicated short lines pick the instance
  nearest the matched blocks of their canonical neighbors
- `recon.go` - Reconciliation pass: anchor-gated post-loop rescue of
  unfound fragments (incl. single-word lines at ReconMinExactAnchors=1)
- `distance.go` - Calculates spatial distance between blocks based on reading order
- `line_server.go` - Manages canonical text lines and handles splitting
- `normalize.go` - Text normalization (lowercase, punctuation removal, Unicode handling)

**`pkg/engines`** - OCR engine registry and readers
- `engines.Engine` interface (Name + Read) maps an engine name to parsing
  code; `engines.Register` adds custom engines (Textract, Document AI, …)
  at init time — built-ins self-register in `builtin.go`
- Each built-in engine has its own package (tesseract, easyocr, apple)
  reading native JSON into engine-specific block types
- `blocks.go` is the engine-neutral format: a JSON array of pre-normalized
  blocks, usable from any OCR source without writing Go
- The adapter contract (word granularity, 0-1 coords, non-empty Extractor,
  Index = emit order) is documented on `ocr.Block`

**`pkg/api`** - Request/response handling
- `SortRequest` - Input validation and parsing
- `SortResponse` - Structured output (document, unhandled lines, meta stats/source); build via `NewSortResponse`
- Delegates to engine readers and normalizers

**`cmd/gollate`** - Command-line tool
- Flag-based CLI for sorting OCR data
- Supports text and JSON output formats
- Optional debug mode with visualization images
- Main entry point for standalone usage

### Data Flow

1. CLI flags or library API call provides engine type, canonical lines, OCR JSON, and page dimensions
2. `api.SortRequest.Parse()` validates and uses engine reader to parse OCR JSON
3. Engine-specific blocks are normalized to common `norm.Block` format
4. `sorters.NewOcrSorter()` creates sorter with normalized blocks and canonical text
5. `sorter.Sort()` performs multi-pass pathfinding to match blocks to lines
6. Results are returned as `api.SortResponse` (document format + unhandled + meta); the CLI defaults to JSON
7. CLI tool outputs results as text or JSON; library consumers use the response directly

### Line IDs

Recent work has added `LineId` tracking to blocks. Each OCR block now tracks which line it belongs to in the original OCR output:
- Tesseract blocks use `LineNum` from the OCR data
- Apple blocks use `LineNum` from the OCR data
- EasyOCR does not currently populate line IDs

This enables downstream consumers to understand which blocks were originally grouped together by the OCR engine.

### Apple Vision OCR (macOS only)

The Apple Vision integration provides native macOS OCR capabilities through the Vision framework:

**Implementation Location**: `pkg/ocr/apple/`
- `apple_vision.m` - Objective-C wrapper for Apple Vision framework
- `engine.go` - Go interface with cgo bindings
- `parse.go` - JSON parsing from Vision framework output
- `types.go` - Data structures (Line, Word, Rect)

**OCR Utility Tool**: `cmd/ocr-util/main.go`
- Command-line tool to OCR image files using Apple Vision
- Usage: `go run cmd/ocr-util/main.go <image-file>`
- Outputs JSON file with naming pattern: `{basename}-ocr.json`
- Example: `go run cmd/ocr-util/main.go image.png` creates `image-ocr.json`

**Tall-image slicing** (`pkg/slicing`, used by both `ocr-util` and `tesseract-util`):
- Apple Vision internally downscales tall images, destroying small text
  (measured: 3% fine-print recall on a 17576px-tall page vs 85% on a crop).
- Images taller than 4000px are automatically sliced into ~1500px strips via
  `internal/imageslicer` (content-aware: cuts only through blank rows),
  OCR'd per strip, and merged back into full-page coordinates.
- Flags on both utils: `-slice` (default true), `-slice-threshold` (4000),
  `-slice-height` (1500). Standard 2112px test pages are never sliced.
- The slicer was inlined from the sibling image-slicer project into
  `internal/imageslicer` so the module has no unpublished dependencies and
  can be imported downstream.

**Engine Adapter**: `pkg/engines/apple/`
- Converts Apple Vision JSON to normalized `Block` format
- Preserves line-level grouping through `LineNum` tracking
- Coordinates are in 0-1 normalized range (consistent with other engines)

**Build Requirements**:
- macOS only (uses Foundation and Vision frameworks)
- Files use `//go:build darwin` constraint
- cgo required for Objective-C interop

### Usage Modes

The library can be used in two ways:
- **CLI tool** - Command-line binary for standalone usage and testing (in `cmd/gollate/`)
- **Go library** - Import `pkg/api` and `pkg/sorters` to use in other Go projects

### Logging

The library uses an internal logger interface (`internal/logger`) to decouple from specific logging implementations:

- **Logger interface** - Defines methods for Debug, Debugf, Error, Errorf, Fatal, and WithError
- **Logos adapter** - Built-in adapter for the goodblaster/logos package (`logger.NewLogos()`)
- **Noop logger** - Silent logger that discards all output (`logger.Noop{}`)
- Library consumers can provide their own logger implementation

The `NewOcrSorter` function accepts a logger parameter. If nil is passed, it defaults to a noop logger.

## Key Insights

### Block Reuse Prevention
Once a block is matched to a line, it's removed from the mapping to prevent it being matched again. This is critical for accuracy.

### Line Splitting
When lines can't be matched (due to missing words or too many permutations), they're split:
1. First by absent words (words in canonical text not found in OCR)
2. Then by sentence boundaries (`. `, `! `, `? `)

The algorithm makes multiple passes, attempting to match progressively shorter lines.

### Distance Calculation
Spatial distance between blocks is the primary scoring metric. Consecutive words in reading order should be spatially close. See `pkg/sorters/distance.go` for the specific formula.

### Hyphenated Words
When `_SplitHyphenatedWords` is enabled, hyphenated words are split into separate blocks with proportional bounding boxes. This improves matching accuracy.

### Output Formats
`sorter.Document()` is the primary output (pkg/sorters/document.go),
modeled on Google Document AI: one readable text blob plus paragraph/token
layout referencing byte spans of it — tokens carry coordinates, not text.
`Document`/`Paragraph`/`Token` have `String()` debug helpers returning
their text slice (unexported backrefs, never serialized). The legacy
`Sort()` return keeps the older Textract-style convention: a flat block
list with empty `Block{}` elements as line-break separators (deprecated in
`api.SortResponse` but preserved for compatibility).
